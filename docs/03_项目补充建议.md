# 项目补充建议

> 项目:商品成分扫描分析 APP
> 日期:2026-07-08
> 用途:对框架与可行性分析的补充,覆盖易被忽略的关键点

---

## 一、产品层面的补充

### 1.1 "真实性"维度需要更具体的定义
当前"看清商品真实性"偏抽象,建议拆解为可落地的核验项:

| 核验维度 | 具体内容 | 数据来源 |
|---|---|---|
| 成分一致性 | 标注成分 vs 备案成分 | 药监局备案 |
| 宣传一致性 | "无硅油/无添加/纯天然"等宣传词 vs 实际成分 | 商品包装 + OCR |
| 浓度排序 | 成分表是否按规范降序排列 | GB 5296.3 |
| 禁用成分 | 是否含违禁成分 | 化妆品安全技术规范 |
| 过期/批号 | 生产日期、批号核验 | 包装 OCR |

> 建议 MVP 先做"宣传一致性"一项,最具话题性和用户感知度。

### 1.2 结果呈现需避免"一刀切"评分
- 单一总分容易误导(如某洗发水含 SLS 但其他都好,总分被拉低)
- 建议:**总分 + 维度分 + 适配人群标签**(如"敏感肌慎用"、"孕产可用")
- 给出"在 XX 品类中排名前 X%"的相对位置,比绝对分数更直观

### 1.3 加入"为什么"的解释链路
- 用户看到"月桂醇硫酸酯钠 - 慎用"会问:为什么?
- 每个结论需可点击展开:成分作用、风险机理、文献来源、适用人群
- 这是建立信任的关键,也是与竞品的差异点

### 1.4 反向场景:扫码查不到怎么办
- 条形码查不到 → 引导拍照 OCR
- OCR 也识别不出 → 引导手动输入成分名
- 成分库未收录 → LLM 兜底 + 标注"待补充"
- **不能让用户卡在"无结果"页面**

## 二、技术层面的补充

### 2.1 OCR 后处理是关键
单纯 OCR 不够,必须做后处理:
- **分词**:配料表无标点时需按成分词典切分("水月桂醇硫酸酯钠甘油" → 水 / 月桂醇硫酸酯钠 / 甘油)
- **同义词归一**:SLS / K12 / 月桂醇硫酸酯钠 / Sodium Lauryl Sulfate → 同一 ID
- **纠错**:OCR 易把"酯"识别成"脂"、"醇"识别成"醉"
- **排序还原**:按成分表原始顺序存储(浓度排序有法律意义)

### 2.2 LLM 调用必须做缓存与降级
- 同一成分组合的扫描结果应缓存(成分组合 hash → 结果)
- LLM 服务不可用时,降级为规则引擎结果
- 设置超时(3-5 秒),避免用户等待过久

### 2.3 离线能力
- 移动端 APP 应内置常见成分库(本地 SQLite),无网也能查
- 仅在需要 LLM 深度分析时才请求云端

### 2.4 图片隐私
- 扫描图片可能含个人信息(收件人、订单号)
- 上传前做裁剪引导或自动检测隐私区域
- 图片用完即删,不留存原图

## 三、数据层面的补充

### 3.1 成分库的字段设计建议
```
成分 ID (主键)
中文名 / 英文名 / INCI 名 / 俗称 / CAS 号
品类(化妆品/食品/日化)
分类(表面活性剂/防腐剂/香精/活性物/...)
功能描述
风险等级(安全/注意/慎用/规避)
风险说明
致敏性 / 致痘性 / 内分泌干扰 / 孕产慎用
适用人群限制
数据来源(国标/文献/权威机构)
更新时间
```

### 3.2 数据版本与溯源
- 每条成分结论必须可溯源(标注来源 + 版本 + 日期)
- 法规变更时,旧结论需标记"已更新"
- 用户历史扫描结果保留当时的版本(避免结果变化引发困惑)

### 3.3 众包数据质量控制
- 用户纠错需投票/审核机制,避免恶意篡改
- 高信誉用户(多次有效贡献)的纠错权重更高
- 建立贡献者积分体系

## 四、运营与增长层面的补充

### 4.1 冷启动获客
- **内容种草**:小红书/B站做"成分揭秘"短视频,带产品试用
- **KOL 合作**:成分党博主、测评类 UP 主
- **场景植入**:与有机超市、母婴店合作,现场扫码立减
- **争议营销**:针对热门商品做"真相揭秘"(注意法律边界)

### 4.2 留存设计
- 扫描历史 + 收藏清单(回头查)
- "我的关注成分"提醒(某成分有新研究时推送)
- 同类商品比价/比成分清单
- 个人健康档案(过敏史、肤质)

### 4.3 数据飞轮
```
更多用户扫描 → 更多商品/成分数据 → 分析更准 → 更多用户
                ↓
            用户纠错反馈 → 数据质量提升 → 信任度上升
```
- 早期需人工补数据,后期靠用户贡献

## 五、法律合规补充

### 5.1 必备文档
- [ ] 隐私政策(个人信息收集、使用、共享)
- [ ] 用户协议(服务边界、免责条款)
- [ ] 免责声明("本应用分析结果仅供参考,不构成医疗或消费建议")
- [ ] 数据来源声明(各成分结论的出处)
- [ ] 品牌方申诉与更正流程

### 5.2 文案红线
- ❌ 不可:"有毒"、"致癌"、"假货"、"骗子"
- ✅ 可以:"含 XX 成分,据 XX 文献可能引起 XX 反应"、"与备案信息不一致"
- ❌ 不可:"最安全"、"100% 无害"
- ✅ 可以:"在同类中排名 X"、"未检出 XX 成分"

### 5.3 资质准备
- APP 上架需 ICP 备案、应用市场资质
- 涉及食品/化妆品信息,可能需增值电信业务许可证
- 如涉及交易,需电商资质(早期避开)

## 六、商业化补充

### 6.1 早期不要碰的模式
- ❌ 付费下载(国内市场不友好)
- ❌ 广告太多(毁体验、毁信任)
- ❌ 接品牌方软文(丧失公信力)

### 6.2 推荐的早期变现
- **导购佣金**:推荐"无 XX 成分"的替代品,带京东/淘宝客链接
- **数据报告**:向品牌方提供行业成分趋势报告(脱敏)
- **API 服务**:给小程序、电商提供成分查询接口

### 6.3 公信力是生命线
- 评分逻辑必须公开透明(Yuka 公开了评分规则)
- 接受品牌方质疑与申诉
- 不接受"花钱改分"
- 一旦失去公信力,产品即死

## 七、需要你重点决策的问题

在正式启动前,建议明确以下决策:

1. **品类范围**:MVP 先做化妆品?食品?还是日化(洗发水/牙膏)?
   - 建议:先做**日化洗护**(你的痛点源头,且竞品少)
2. **平台优先级**:Web / 小程序 / APP 的顺序?
   - 建议:Web 验证 → 小程序获客 → APP 沉淀
3. **数据策略**:自建成分库 vs 直接用 LLM?
   - 建议:核心 500 种自建,其余 LLM 兜底
4. **是否引入账号体系**:MVP 要不要登录?
   - 建议:MVP 免登录,降低门槛;后期再补
5. **OCR 方案**:开源 PaddleOCR vs 云服务?
   - 建议:Web MVP 用云服务(快、准);APP 阶段评估自建
6. **LLM 选型**:DeepSeek(便宜)/ 通义(中文好)/ GPT(强)?
   - 建议:DeepSeek 起步,成本低、中文够用
